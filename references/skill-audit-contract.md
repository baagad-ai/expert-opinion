# Skill Audit Output Contract

**Status:** Canonical — field names defined here are the authoritative API between all
phases of the skill audit pipeline. All executors must use these exact names.

**Schema version:** `1.0`

**Changelog:**
- `1.0` — initial versioned release; added `schema_version` field, `failed_roles` and `timed_out_roles` to `SkillAuditMetadata`.

---

## S01 → S02: SkillAuditIntakePackage

```typescript
interface SkillAuditIntakePackage {
  schema_version: "1.0";           // Required — activates version-check in S02-1 consumer
  skill_path: string;              // Absolute or relative path to skill root directory
  skill_files: SkillFile[];        // All files loaded during intake
  structural_scan: StructuralScan; // Pre-validation results
  confirmed_roles: SkillAuditRole[];
  inferred_context: SkillContext;
}

interface SkillFile {
  path: string;                    // Relative path from skill root (e.g. "SKILL.md")
  role: "router" | "workflow" | "reference" | "template" | "script" | "other";
  content: string;                 // Full content or key excerpts (max 2000 tokens per file)
  line_count: number;              // Actual line count (not estimated)
  truncated: boolean;              // true if content was truncated due to size
}

interface StructuralScan {
  skill_name: string;              // From YAML frontmatter name field
  has_frontmatter: boolean;        // SKILL.md has valid YAML frontmatter
  frontmatter_valid: boolean;      // name + description fields present
  line_count_skill_md: number;     // Actual lines in SKILL.md
  exceeds_500_lines: boolean;      // K002 violation flag
  has_markdown_headings: boolean;  // K001 violation flag (# in SKILL.md body)
  workflow_count: number;          // Number of files in workflows/
  reference_count: number;         // Number of files in references/
  template_count: number;          // Number of files in templates/
  has_output_contract: boolean;    // Any file matching *contract* or *schema* in references/
  dispatches_subagents: boolean;   // "subagent" or "parallel" appears in any workflow file
  accepts_user_input: boolean;     // "ask_user_questions" or intake phase present
  calls_external_services: boolean; // fetch_page or search-the-web in any file
  violations: StructuralViolation[];
}

interface StructuralViolation {
  rule: string;                    // e.g. "K001", "K002", "missing-frontmatter"
  file: string;                    // Which file triggered the violation
  detail: string;                  // Human-readable description
  severity: "blocking" | "major" | "minor";
}

interface SkillAuditRole {
  role: string;                    // Display name from skill-audit-roles.md
  domain: string;                  // Expert domain
  focus_questions: string[];       // 2–4 questions specific to THIS skill's content
  rationale: string;               // Why this role applies to this specific skill
  is_mandatory: boolean;           // From Role Selection Heuristics table
  research_queries: string[];      // 2–4 pre-approved web search queries generated by the
                                   // orchestrator during S01 intake — before skill files are
                                   // passed to subagents. Subagents may ONLY search using
                                   // these queries; they must not derive queries from skill content.
}

interface SkillContext {
  skill_purpose: string;           // What the skill does (from description field or inferred)
  intended_user: string;           // Who uses this skill (from content or inferred)
  complexity_tier: "simple" | "router" | "enterprise";
    // simple: single SKILL.md, no workflows
    // router: SKILL.md + workflows/ + references/
    // enterprise: multi-workflow + output contracts + subagent dispatch
  maturity_estimate: string;       // Initial estimate: "prototype" | "v1" | "production"
}
```

## S02 → S03: SkillAuditResearchPackage

```typescript
interface SkillAuditResearchPackage {
  reports: SkillAuditReport[];
  structural_scan: StructuralScan; // Passed through from S01 for S03 use
  metadata: SkillAuditMetadata;
}

interface SkillAuditReport {
  role: string;                    // Matches SkillAuditRole.role exactly
  domain: string;                  // Matches SkillAuditRole.domain exactly
  report: string;                  // Full text of the filled skill-expert-report.md template
  missing_sections: string[];      // Empty [] if all required sections present
  finding_count: number;           // Number of <finding> blocks in the report
  highest_severity: "critical" | "major" | "minor" | "informational";
  confidence_score: number;        // 0.0–1.0: expert's confidence in their findings
                                   // (drawn from <confidence_score> in the report)
}

interface SkillAuditMetadata {
  skill_path: string;
  skill_name: string;
  total_roles: number;
  completed_roles: number;
  failed_roles: string[];    // role names that returned NO output (for targeted re-dispatch)
  timed_out_roles: string[]; // role names cut off by timeout (subset of failed_roles)
  inferred_context: SkillContext;  // Passed through from S01
  structural_scan: StructuralScan; // Passed through from S01
}
```

## S03 Output: SkillAuditDocument

The final synthesis document emitted as a markdown file.
Filename pattern: `skill-audit-{skill-name}-{YYYY-MM-DD-HHmm}.md`

Required sections (in order):
1. `<overview>` — skill identity, maturity score, overall health
2. `<structural_health>` — pre-validation results (from StructuralScan)
3. `<maturity_scorecard>` — 5-dimension quantitative scorecard
4. `<per_role_highlights>` — one row per expert
5. `<cross_cutting_findings>` — convergent findings from 2+ experts
6. `<contradictions>` — expert disagreements (omit if none)
7. `<prioritized_recommendations>` — ranked action table
8. `<remediation_plan>` — phased fix schedule
9. `<open_questions>` — unresolved questions
10. `<incomplete_coverage>` — partial reports (omit if none)

## Emission Rules

The audit intake workflow MUST:
1. Emit `SkillAuditIntakePackage` only after user has confirmed the role list.
2. Always include `structural_scan` — it runs automatically before role proposal.
3. Always set `is_mandatory` per the Role Selection Heuristics table in `skill-audit-roles.md`.
4. **Emit `schema_version: "1.0"` as the first field.** Without this field, the version-check in S02-1 is never triggered and schema drift produces silent consumer failures.

The research workflow MUST:
1. Emit `confidence_score` drawn from the `<confidence_score>` field in each expert report.
2. Emit `finding_count` and `highest_severity` as computed fields (not from the template).
3. Never repair missing sections — record in `missing_sections` and pass through verbatim.

The synthesis workflow MUST:
1. Complete the full pre-synthesis analysis pass (convergence, contradiction, severity
   aggregation, open questions merge) before filling any template section.
2. Compute `maturity_scorecard` from the aggregated finding data.
3. Emit `<remediation_plan>` with explicit phases tied to `<prioritized_recommendations>` ranks.

## Partial-Retry Recovery

When `metadata.failed_roles` is non-empty, do NOT re-run the full pipeline. Instead:

1. Extract the failed role entries from the original `SkillAuditIntakePackage.confirmed_roles`
   where `role` matches any name in `failed_roles`.
2. Re-construct subagent tasks for those roles only (same task template as S02-2).
3. Dispatch in parallel. Collect outputs.
4. Merge new reports into the existing `SkillAuditResearchPackage.reports` (replace the
   empty slot for each recovered role; update `completed_roles` and clear the role from
   `failed_roles`).
5. Proceed to S03 synthesis with the merged package.

The same pattern applies to `ResearchOutputPackage.metadata.failed_roles` in the general pipeline.

## Version Compatibility

Consumers MUST check the `schema_version` at the top of this file. If a consumer was written
against version `X` and this file declares version `Y > X`, emit a structured warning:
```
WARNING (skill-audit-contract): consumer built against schema 1.0, current schema is {Y}.
Check for removed or renamed fields before proceeding.
```

## Breaking vs. Non-Breaking Contract Changes

**Breaking changes** (require major version bump — all consumers must update):
- Removing any field that is currently required
- Renaming any existing field
- Changing the type of any existing field
- Adding a new required field (consumers will fail on missing field)
- Changing a field from optional (`?`) to required

**Non-breaking changes** (minor version bump — existing consumers continue to work):
- Adding a new optional field (`?`) with a documented default or omission behavior
- Adding informational fields to `SkillAuditMetadata`
- Adding a new `complexity_tier` or `maturity_estimate` value, if consumers handle unknown enum values

**Version bump convention:**
- Major bump (e.g. `1.0` → `2.0`): breaking change — update `schema_version` in this file AND in all consuming phases (S02-1 pre-hook, S03-1 pre-hook)
- Minor bump (e.g. `1.0` → `1.1`): non-breaking — update `schema_version`; existing consumers continue to function

**Migration guide template** (include in the `Changelog` section when bumping):

```
## Migration: schema {X.Y} → {X+1.0}

### Removed fields
- `old_field_name` (was in `InterfaceName`) — replace with `new_field_name` or derive from `{other_field}`

### Added required fields
- `new_field_name: type` (in `InterfaceName`) — populate from `{source}`; backwards-compat default: `{default}`

### Consumer update checklist
- [ ] Update `schema_version` check in audit-skill.md S02-1 pre-hook
- [ ] Update `schema_version` check in audit-skill.md S03-1 pre-hook
- [ ] Update field references in workflows/audit-skill.md
- [ ] Re-run scripts/validate.sh
```
